1. 프로젝트 개요
프로젝트명: TravelPlan (여행 계획 및 예약 사이트)

기획 및 개발: 박진

개발 기간: 2026.07.01 ~ 2026.07.08 (총 1주일)

한 줄 소개: 사용자가 원하는 여행지를 탐색하고, 일정 관리부터 숙소/식당 예약 및 가상 결제까지 한 번에 처리할 수 있는 정적 웹 서비스 prototype

2. 기획 배경 및 목적
복잡한 여행 준비 과정의 단일화: 여행지 검색, 일정 짜기, 숙소 및 식당 예약을 각각 다른 플랫폼에서 처리해야 하는 번거로움을 해결하고자 하나의 플랫폼에서 유기적으로 이어지도록 기획함.

배포 인프라 다원화 연습: 프론트엔드와 백엔드 API를 분리해 각각 호스팅 환경에 배포해 보고, 실제 상용 서비스에 가까운 배포 아키텍처를 경험함.

사용자 경험(UX) 중심의 UI 구현: 파스텔 계열의 밝고 깔끔한 톤을 사용하여 사용자가 여행을 계획할 때 느낄 수 있는 설렘을 시각적으로 극대화함.

3. 기술 스택 (Tech Stack)
Frontend: HTML5, CSS3, Vanilla JavaScript (Vanilla JS 기반 정적 페이지 아키텍처)

Backend (Mock API): json-server (REST API 기반 데이터 모킹)

Deployment (배포):

Frontend: Netlify (정적 호스팅 및 커스텀 도메인 연결)

Backend: Render.com (json-server 구동 및 API 서버 호스팅)

5. 프로젝트 핵심 성과 및 도전 과제
① 백엔드 API 분리 배포 성공
정적 파일만 호스팅 가능한 Netlify의 제약을 넘기 위해 Node.js 엔진 기반의 json-server를 Render.com에 단독 배포하여 CORS 이슈를 해결하고 프론트엔드와 유기적인 fetch 통신 환경을 구축함.

② 클라이언트 사이드 데이터 최적화
전체 공유가 필요한 여행지 및 추천 인프라 데이터는 백엔드 API(db.json)에서 호출하고, 개별 사용자의 예약 상태 및 세션 정보는 브라우저 저장소(LocalStorage)를 적극 활용하여 불필요한 서버 트래픽을 줄이고 실시간성을 확보함.

6. 트러블슈팅 (Troubleshooting)
[문제 상황: 카카오 로그인 OAuth 2.0 연동 실패]
현상: 배포 환경(Netlify)에 커스텀 도메인을 연결한 뒤 카카오 로그인 연동을 시도했으나, 주소를 계속 수정해도 KOE205(동의 항목 누락) 및 KOE006(등록되지 않은 Redirect URI) 에러가 발생하며 제자리걸음을 반복함.

원인 분석:

설정 오판: KOE205는 카카오 개발자 센터 내부에서 이메일/닉네임 수집 동의 항목을 누락해 발생한 권한 문제였으나, 도메인 주소 문제로 오판하여 URI 세팅만 반복 수정함.

구조적 한계: OAuth 2.0 REST API 방식은 인가 코드를 받아 토큰을 교환해 줄 '진짜 백엔드 로직'이 필수적임. 그러나 사용된 json-server는 가짜 Mock API 서버이기 때문에 복잡한 인증 보안 로직을 소화할 공간이 없었음.

해결 및 개선 방향: 제한된 제출 마감 일정 내 안정성을 확보하기 위해 로컬 로그인 구조로 안전하게 우회 처리함. 차후 카카오 자바스크립트 SDK를 이용한 팝업창 인증 방식을 도입하여 백엔드 개입 없이 프론트엔드 단독으로 토큰을 다루도록 리팩토링할 예정임.

7. 기대 효과 및 향후 발전 계획
사용자 맞춤형 고도화: 임시 수첩 역할인 LocalStorage 환경을 고도화하여 사용자별 독립적인 완전한 데이터 적재 인프라 구축.

진짜 백엔드(Node.js + Express) 도입: json-server를 완전히 걷어내고 Express와 가상 DB 인프라를 직접 구축하여 완벽한 OAuth 2.0 소셜 로그인 및 공용 게시판 기능을 리팩토링할 계획임.
